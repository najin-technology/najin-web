# 네이버 블로그 전수 마이그레이션 + 콘텐츠 IA 재정비 — 설계

작성: 2026-06-16 · 승인: 사용자 (Approach ① 태그 기반 사례 스트림 + 역할 분리)

## Context

나진테크 공식 사이트의 게시글은 네이버 블로그(`blog.naver.com/kinghak1`, 31개)에서 일부만(8개) 옮겨졌고,
본문은 요약본·이미지는 placeholder/깨짐(2건)이 섞여 있다. 또한 콘텐츠 종류 4개(제작사례 글 `posts` /
거래처 `customers` / 납품실적 `client_deliveries` / 제품 `products`)의 **경계가 정의되지 않아** 같은 작업이
어디로 가야 할지 모호했다.

목표: (1) 블로그 31개를 **전부** 적절한 위치로 마이그레이션(이미지·본문 ground truth = 블로그),
(2) 콘텐츠 종류의 **역할을 명확히** 분리, (3) SEO(31개 아카이브) + 전면 노출 엄선(featured) 둘 다.

## 핵심 결정 — 경계 규칙

> **보여주고 싶으면 사례글(post), 사실 기록이면 실적연혁(delivery).**

| 종류 | 정체성 | 사진 | 비고 |
|---|---|---|---|
| `post` (사례글) | "이런 걸 **만들었다**" 시각 증거 | 필수 | 블로그 가공/제품 글 → 여기 |
| `client_delivery` (실적연혁) | "이 거래처와 **언제부터 뭘** 거래" 사실 | 불필요 | 거래처 한 줄, 유지 |
| `customers` | 거래처 프로필 | — | 변경 없음 |
| `products` | 사업영역(공정) 쇼케이스 | — | 변경 없음 |

## 데이터 모델 변경

`posts` 테이블에 컬럼 2개 추가 (migration `027_posts_process_featured.sql`):
- `process_category text` — 우레탄 · 합성수지 · CNC · 금형 · EV · 기타 (`products`의 5분류 재사용). **"제품별 태깅"의 핵심 분류축.**
- `featured boolean default false` — 전면 노출(홈·포트폴리오) 큐레이션 플래그.
- 기존 유지: `category`(제작사례/제품), `tags`(자유), `customer_id`(optional), `image_urls`, `original_date`.

## 파생 뷰 (태그/플래그에서 자동 — 중복관리 제거)

- `/posts` — 전체 + **공정·태그·거래처 필터** (이미 `?tag=` 작동, 공정 필터 추가)
- `/portfolio` — `featured=true` + 공정 필터
- `/clients/[slug]` — 그 거래처(`customer_id`) 연결 사례글 갤러리 **+** `client_deliveries` 타임라인
- `/business` — 공정 5종, 각 공정의 대표 사례글
- 홈 — `featured` 사례글

## 블로그 31개 라우팅

- **사례글 `post` (~26)** — 공정 분류:
  - 우레탄(11): 배관받침대·케스팅·디바이드×3·pad금형·A95×3·D95·성형제품·합판코팅·제전방지·TANK PAD
  - 합성수지(7): PE환봉·PE디바이드·연녹PC·아세탈·신주·합성수지가공품·제품소개
  - CNC(5): 3D MC×2·엔진케리어MC·자동차부품안착·SUS후렌지
  - 금형(2): AL금형·AL가공품
  - 거래처 연결: **알려진 것만**(SK·GM·르노삼성·현대 등), 나머지는 거래처 없이 공정만. 창작 금지.
- **회사소식 `notices` (~5)** — 사례 아님: 회사소개·확장이전·이전안내·견적문의·나진테크 현장
- 각 글: 블로그 실사진 다운로드 + 본문(A 이미지재싱크 · B 본문보강 반영). 기존 8개도 동일 기준 재정비.

## 마이그레이션 실행 방식 (중요)

- **이미지**: 블로그 실사진 → `/public/images/products/`(또는 `/posts/`) 다운로드, 로컬 경로 참조. (블로그 #6 방식 검증됨)
- **본문/메타 DB 쓰기**: 거래처 실명 등 민감정보 포함 → **#70 정책상 git seed 아닌 Supabase DB INSERT/UPDATE로** (supabase MCP). 일반(비민감) 콘텐츠는 seed 가능하나, 일관성 위해 DB 직접 권장.
- 모든 운영 DB 쓰기는 **사용자 승인 후** 실행.

## 관리자 UX

post 편집기에 필드 추가: `process_category`(드롭다운) · `featured`(체크) · `customer`(선택). 어르신용:
"사진 올리고 → 공정 고르고 → (알면) 거래처 고르고 → 잘 나온 건 featured". 후속 단계.

## 단계 (phasing)

1. **스키마** — `027` 마이그레이션 (`process_category`, `featured`)
2. **콘텐츠 마이그레이션** — 31개 크롤 → 26 사례글 + 5 공지. 이미지·본문·태그·거래처·공정 세팅 (DB 쓰기, 승인 필요)
3. **뷰 연결** — posts 공정 필터, 거래처 사례 자동표시, business/portfolio/홈 featured·공정 반영
4. **관리자 UI** — 편집기 필드 (후속)

## 비범위 / 가드

- 거래처 실명: 코드·GitHub·seed에 박지 않음. DB에만. 알려진 것만, 창작 금지(#70).
- `client_deliveries` 폐지 안 함 — 역할 분리로 공존.
- 1~2단계 먼저, 4단계(UI)는 후속으로 분리 가능.
- 운영 DB 쓰기·커밋·배포는 사용자 승인 필수.
